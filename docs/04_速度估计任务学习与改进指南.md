# 速度估计任务学习与改进指南

## 1. 当前任务目标

你负责的不是“继续做滤波页面”，而是：

> 从代码层真正把当前速度估计方法读懂，说明它怎么工作、可能哪里有缺陷、哪些点值得改进。

## 2. 当前基线方法是什么

当前基线是：

- `State Observer + PLL`
- 无感速度估计

不是：

- 编码器测速
- Hall 传感器测速

## 3. 你要输出什么

建议最终形成下面四类成果：

1. `STO + PLL` 代码链路说明
2. 估计方法缺陷分析
3. 一个或多个“可落地的小改进思路”
4. 若改代码，则给出最小补丁 + 验证方式

## 4. 学习路径

### 第一步：先搞清配置层

看：

- `firmware/current_project/first_try.wb_def`

确认这几个点：

- 为什么当前不是 Hall/编码器
- 为什么当前选的是 `STATE_OBSERVER_PLL`
- 生成工程后哪些对象会在 `mc_config.c` 和 `mc_tasks.c` 出现

### 第二步：再搞清运行主流程

看：

- `firmware/current_project/Src/mc_config.c`
- `firmware/current_project/Src/mc_tasks.c`

你需要回答：

- `STO_PLL_M1` 什么时候初始化
- 平均机械速度在哪里算出来
- 这个速度什么时候被认为可靠
- 什么时候速度环从虚拟速度切换到观测器速度

### 第三步：再看库内部

重点看：

- `sto_pll_speed_pos_fdbk.c`
- `sto_pll_speed_pos_fdbk.h`
- `sto_speed_pos_fdbk.h`
- `speed_pos_fdbk.c`
- `speed_pos_fdbk.h`

你不一定要一开始把全部数学都推完，但至少要弄清：

- 哪些量是状态量
- 哪些量是观测器输入
- 哪些函数负责角度
- 哪些函数负责速度
- 哪些函数负责可靠性判定

## 5. 当前可以重点思考的缺陷

### 低速问题

无感估计在低速时通常更难，因为反电势弱，信号信噪比差。

### 参数依赖

状态观测器依赖电机模型参数，如果参数不准，估计质量会下降。

### 动态阶段误差

启动、突加速、突减速时，估计值可能出现超调、滞后或短时不稳定。

### 切换阶段问题

从虚拟速度过渡到观测器速度时，可能存在收敛过程或切换抖动。

### 可靠性判定保守或激进

如果“可靠”判得太早，可能把不稳定估计值提前送入闭环。  
如果判得太晚，又会拖慢进入正常控制的时机。

## 6. 可尝试的改进方向

建议优先从低风险方向开始：

1. 先做代码注释与链路梳理，不改逻辑
2. 先做实测 CSV 回放分析，理解估计值动态特性
3. 先尝试观测器输出后的轻量后处理
4. 再考虑估计器内部阈值、平均策略、可靠性策略等小改动
5. 最后才考虑更换方法或大改结构

## 7. 当前不建议直接做的事

- 直接把 `STO + PLL` 换成另一套全新算法
- 大改多个库文件再一次性提交
- 只凭理论觉得“应该更好”就改，不做对比验证

## 8. 验证方式建议

每次改动尽量走这个顺序：

1. 先写文字说明改动意图
2. 对比 `baseline_project` 和 `current_project`
3. 如果是估计输出相关改动，先做离线分析或仿真
4. 再做板上实验
5. 记录现象和风险

## 9. 很关键的一句话

你这条线真正加分的地方，不在于“把估计器彻底重写”，而在于：

- 你能准确指出当前估计方法的代码入口
- 你能把它的原理和局限讲清楚
- 你能提出一个真实、合理、可验证的小改进
