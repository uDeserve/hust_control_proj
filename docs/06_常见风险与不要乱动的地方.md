# 常见风险与不要乱动的地方

## 1. 不要随便重新生成工程

`first_try.ioc` 和 `first_try.wb_def` 一旦重新生成，可能覆盖现有代码、改掉配置或让工程结构变化。  
除非明确要做配置层修改，否则不要点生成。

## 2. 不要把 patch 目录当主工程

`patches/current_patch` 只是快照。  
如果只改这里，不改 `firmware/current_project`，最后主工程不会得到真正更新。

## 3. 不要把 build 产物提交进仓库

尤其是：

- `*.o`
- `*.axf`
- `STM32CubeIDE/Debug/`
- 各种日志

这些文件会让仓库变臃肿，也会污染 diff。

## 4. 不要一上来大改估计器

速度估计方法本体属于控制链核心逻辑。  
一上来大改很可能出现：

- 无法编译
- 启动失败
- 切换异常
- 转速不稳

更合理的方式是：

- 先读懂
- 先分析
- 先做小改
- 先保留回退路径

## 5. 不要把“滤波问题”和“估计问题”混成一个结论

当前工程里两者都存在，但层级不同：

- 速度估计：`STO + PLL`
- 速度滤波：在估计输出之后做的处理

你在分析现象时必须尽量区分：

- 这是估计器本体的问题
- 还是滤波/闭环造成的表现差异

## 6. 不要忽视切换阶段

很多异常不在稳态，而在：

- 启动初期
- 观测器收敛阶段
- 虚拟速度切到观测器速度阶段
- 停转尾段

如果只看稳态，很容易漏掉最关键的问题。

## 7. 不要跳过文档

后续答辩很看重真实性。  
所以每个关键判断最好都能留下：

- 代码路径
- 原理说明
- 实验记录
- 仿真或回放依据
